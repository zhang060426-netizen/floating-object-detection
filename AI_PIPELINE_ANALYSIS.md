# AI_PIPELINE_ANALYSIS.md

## 1. AI 链路总览

本系统 AI 能力由四类链路组成：

1. **图片检测链路**：YOLO 识别 + CLAHE 增强 + Qwen-VL 多模态分析 + 治理建议 + Word 报告。
2. **视频检测链路**：视频帧采样 + YOLO 逐帧检测 + 关键帧/结果视频输出。
3. **实时检测链路**：摄像头帧读取 + YOLO 实时推理 + ByteTrack 跟踪 + 会话记录。
4. **训练/评估链路**：YOLO 数据集 + 训练脚本 + 验证/测试指标 + 评估记录。

当前阶段不重新训练模型，不替换大模型 API，不修改类别定义；只做链路梳理和工程化优化规划。

## 2. 图片检测 + 多模态分析链

流程：用户上传图片 → 选择已发布模型 → YOLO 目标检测 → 输出目标框/置信度/类别 → 裁剪目标并保存 → 生成标注图 → CLAHE 增强 → 构造多模态分析上下文 → Qwen-VL/阿里云百炼 API → 污染描述/风险分析/治理建议 → 检测记录入库 → Word 报告导出。

关键输入：

- 原始图片
- 模型 ID / 权重路径
- YOLO 检测结果：类别、置信度、坐标框
- CLAHE 增强图或增强参数
- 多模态分析 prompt

关键输出：

- 标注图、裁剪图
- `detection_result` JSON
- Qwen-VL 分析文本
- 治理建议
- Word 报告
- `detection_records` 与 `detection_crops` 记录

风险点：

- Qwen-VL API 密钥、超时、限流、费用不可控。
- YOLO 漏检会影响多模态分析上下文。
- CLAHE 增强可能改善可视化，但也可能改变图像观感，需要明确是否参与模型输入或仅参与分析/展示。
- 检测结果 JSON 若未稳定，会影响前端展示、报告和历史记录。

## 3. 视频检测链路

当前文档描述为：创建任务后后台/子进程逐帧推理，前端轮询进度。

建议状态机：

| 状态 | 含义 | 可恢复性 |
|---|---|---|
| `pending` | 任务已创建，等待处理 | 可取消 |
| `processing` | 正在抽帧/检测 | 可记录进度 |
| `completed` | 检测完成 | 可展示结果 |
| `failed` | 任务失败 | 应记录错误原因 |
| `cancelled` | 用户取消 | 应清理临时文件 |

优化方向：明确采样帧率与最大处理帧数；长视频增加任务超时与资源释放；关键帧按目标出现、置信度、间隔去重；结果文件增加生命周期清理策略；前端轮询改为可配置间隔。

## 4. 实时检测链路

当前限制：仅支持本地物理摄像头，单摄像头，不支持网络摄像头。

关键依赖：摄像头读取稳定性、模型推理耗时、ByteTrack 跟踪参数、帧率/分辨率/跳帧策略、会话记录与目标去重策略。

| 优化项 | 优先级 | 说明 |
|---|---:|---|
| 模型懒加载与缓存 | P0 | 避免每帧/每次请求重复加载模型 |
| 降采样/跳帧策略 | P1 | 降低 CPU/GPU 压力 |
| 跟踪 ID 去重 | P1 | 减少重复入库 |
| 会话开始/停止可靠性 | P1 | 防止摄像头资源占用 |
| 网络摄像头支持 | P3 | 当前阶段非目标 |

## 5. 训练与评估链路

当前训练目录事实：

- 目录：`1项目代码/floating-objects-detect-web/other/model_train/detect/`
- 脚本：`code/train.py`、`code/val.py`、`code/predict.py`
- 数据：`dataset/small_dataset/data.yaml`
- 权重：`weights/yolov8n.pt`、`weights/yolo11n.pt`、`weights/yolo12n.pt`、`weights/yolo26n.pt`
- 类别：`floating_object`

当前测试集指标：Precision 0.889，Recall 0.827，mAP50 0.915，mAP50-95 0.659。

训练工程化建议：Web 系统负责模型管理、上传、发布、评估；训练目录负责离线训练/验证/预测；文档中明确训练环境、数据结构、输出产物规范。后续若做“训练任务管理”，应作为独立阶段，不与当前规划阶段混合。
